# 保全排班工具 Spec v2

> 版本目的：對齊目前已上線功能（含調班自動修復與測試模式日期覆寫）。

## 1. 產品目標

將每月人工排班時間從約 1 小時降到幾分鐘，並以系統化規則檢查降低違規風險。

## 2. 範圍

### 2.1 In Scope

- 月排班自動產生（6 名人員）
- 台灣行事曆 API 判定上班日/放假日
- 規則檢查與稽核證據（6 條）
- CSV / XLSX / JSON 匯出
- 調班流程（借班 + 還班 + 剩餘班表自動修復）
- UI 測試模式（可指定「今天」日期）

### 2.2 Out of Scope

- 跨月欠班與跨月還班
- 審核流程（主管核准）
- 多組織／多角色權限

## 3. 核心模型

### 3.1 人員

- `id`: string
- `name`: string

### 3.2 哨點與工時

- 平日：`A/B/C`（10h）、`D/E`（12h）
- 假日：`F/G`（12h）

### 3.3 每日需求

- 上班日：`A~E` 共 5 人
- 放假日：`F/G` 共 2 人

### 3.4 跨月狀態（carry_over）

- `consecutive_days`
- `last_post`
- `last_holiday_dow`
- `last_holiday_post`

## 4. 日型判定

### 4.1 預設來源

- `GET /taiwan-calendar/{year}/{month}`（`https://api.pin-yi.me`）
- 以 `isHoliday` 判定：
  - `true` = 放假日
  - `false` = 上班日

### 4.2 失敗處理

- UI：若 API 失敗，產生排班中止並提示錯誤。
- CLI：可用 `--use-input-holidays` 強制改用輸入 JSON 的 `holidays`。

## 5. 規則

### 5.1 硬限制

1. 連續上班不得超過 6 天
2. 同一哨點不可連兩天
3. 假日班須六/日交替
4. 假日哨點須 F/G 交替

### 5.2 公平目標

5. 月工時差距目標：`<= 12h`
6. 每哨點分配差距目標：`<= 1`

## 6. 調班規格（v2）

### 6.1 定義

- 借班日：原值班人改休，代班人上該班
- 還班日：代班人改休，原值班人接回該班

### 6.2 條件

- 借班日與還班日都必須在同一個月
- 借班日與還班日不可同一天
- 兩日期都必須是「可修改日期」（今天與未來）
- 借班日：原值班人需有班、代班人需休息
- 還班日：原值班人需休息、代班人需有班
- 借班班別與還班班別工時必須相同（10h 對 10h、12h 對 12h）

### 6.3 自動修復

- 鎖定借班日與還班日交換結果
- 從兩日期中較早者開始，重排到月底
- 目標：滿足規則且盡量減少變更格數

### 6.4 失敗行為

- 顯示「本次調班失敗」原因
- 提供可行還班日建議（若存在）

## 7. UI 行為

### 7.1 可修改日期

- 預設以系統日期作為「今天」
- `今天以前` 鎖定不可調班

### 7.2 測試模式

- 可在側欄開啟「測試模式：自訂今天日期」
- 只影響 UI 可選日期，不改系統時間

## 8. 輸出

- `schedule_YYYY_MM.csv`
- `schedule_YYYY_MM.xlsx`
- `report_YYYY_MM.json`
  - 含 `violations`
  - 含 `audits`（每條規則門檻、實測值、證據）
  - 調班後含 `adjustment`（變更格數、變更日期）

## 9. 驗收標準

- 基本排班可生成
- 規則檢查結果可視化且可匯出
- 調班成功時能自動修復並維持合規
- 調班失敗時能給明確原因與可行建議
